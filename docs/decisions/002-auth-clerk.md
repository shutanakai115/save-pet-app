# ADR-002: 認証方式として Clerk を採用

## ステータス

採用

## コンテキスト

ユーザー登録・ログイン機能が必要。以下の選択肢を検討した：

1. **自前でJWT認証を実装** - Go側で認証ロジックを実装
2. **Supabase Auth** - Supabaseの認証機能を利用
3. **Clerk** - 認証に特化したSaaS

## 決定

**Clerk** を採用する。

## 理由

### Clerk を選んだ理由

- **実装コストが低い**: コンポーネントを配置するだけで認証UIが完成
- **PWA対応**: モバイルWebでも問題なく動作
- **Next.js との統合が容易**: 公式でサポートされている
- **Go API との連携**: JWTトークン検証でGo側と連携可能

### 他の選択肢を見送った理由

| 選択肢 | 見送った理由 |
|--------|-------------|
| 自前JWT実装 | 学習コストと実装コストが高い、セキュリティリスク |
| Supabase Auth | DB選択がPostgreSQLで確定しており、Supabase全体を使う必要がない |

## 結果

- 認証の実装工数を大幅に削減できる
- セキュリティはClerkに任せ、アプリのコア機能に集中できる
- Go側ではClerkのJWTトークンを検証する形で連携
